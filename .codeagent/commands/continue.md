## 协作任务

根据当前PR状态和用户指令，执行以下任务：

### 第一步：上下文分析
1. **理解当前状态**：分析PR的当前进度和问题
2. **识别待办事项**：从评论和反馈中提取需要处理的任务
3. **评估变更影响**：了解已有变更的影响范围
4. **确定工作重点**：基于反馈和指令确定优先级

### 第二步：问题诊断
1. **代码审查反馈**：处理 reviewer 提出的代码问题
2. **功能完善**：补充遗漏的功能或改进现有实现  
3. **测试覆盖**：增加或完善测试用例
4. **文档更新**：根据代码变更更新相关文档

### 第三步：代码改进
1. **修复已知问题**：解决PR中发现的bug或问题
2. **性能优化**：改进代码性能和效率
3. **代码重构**：优化代码结构和可读性
4. **标准合规**：确保代码符合项目规范

### 第四步：质量保证
1. **功能验证**：确保所有功能正常工作
2. **回归测试**：运行测试套件，确保没有破坏现有功能
3. **代码格式化**：运行linter和格式化工具
4. **构建验证**：确保代码能正常构建和部署

### 第五步：协作沟通
1. **更新说明**：总结本次改进的内容
2. **回应反馈**：针对具体的review comments给出回应
3. **标注变更**：在commit message中清晰说明变更内容
4. **请求复审**：如需要，请求相关人员重新审查

## 处理策略

### 针对不同类型的反馈

**代码质量问题**:
- 重构代码改善可读性
- 优化算法和数据结构
- 添加必要的错误处理

**功能缺陷**:
- 修复bug和逻辑错误
- 补充遗漏的功能点
- 改进用户体验

**测试相关**:
- 增加单元测试覆盖率
- 编写集成测试用例
- 修复测试中的问题

**文档和配置**:
- 更新API文档
- 完善使用说明
- 调整配置文件

### 特殊情况处理

**冲突解决**:
- 如果有merge冲突，优先解决冲突
- 保持与base分支的同步

**大幅修改**:
- 如需要大幅重构，先与团队沟通
- 分步提交，便于审查

**性能影响**:
- 对性能敏感的修改进行基准测试
- 提供性能影响分析

## 输出要求

完成任务后，请提供：

1. **变更摘要**: 本次修改的核心内容和解决的问题
2. **技术说明**: 重要技术决策的说明和理由  
3. **测试结果**: 相关测试的执行结果
4. **后续计划**: 如果还有待完成的工作，说明计划

## 注意事项

- **保持PR专注**: 只处理与当前PR相关的问题，避免功能蔓延
- **及时沟通**: 遇到重大设计决策时主动寻求反馈
- **版本控制**: 创建清晰的commit历史，便于代码审查
- **向前兼容**: 确保修改不会破坏现有功能和API
- **文档同步**: 代码修改后及时更新相关文档
